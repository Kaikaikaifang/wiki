> 真正让我把这次迁移当成“系统工程”而不是“运维操作”的，不是 ClickHouse 本身有多复杂，而是眼前这套生产形态已经明确告诉我：这不是一次适合赌停机窗口的数据库搬家。

我这次梳理的对象，不是一个抽象的“单节点 ClickHouse”，而是一套非常具体的生产约束组合：

- 当前部署形态仍是单实例 StatefulSet，只有一个 ClickHouse Pod 和一个服务入口；
- 核心业务表 `scalar`、`media`、`log` 仍是单机 `MergeTree`；
- 应用与写入管道都直接写逻辑表名，而不是经过分布式入口；
- 历史数据量已经到 `7118 GiB`，这意味着迁移的主要矛盾不再是“怎么建集群”，而是“怎么在长时间回灌期间不丢写、不乱表、不把切换窗口拖成事故”。

在这种起点下，我不建议走“旧库原地改 replicated，再逐步加节点”的路线。那条路在纸面上像是渐进式升级，但对这种数据量来说，本质上只是把一次大迁移拆成两次高风险变更。我更倾向的方案是：**直接并行搭建目标集群，预建最终表形态，用 `Vector` 双写兜住新增写入，再用长时间历史回灌把旧数据追平。**

## 先说结论

这次迁移方案的核心决策应该固定为：

- 目标拓扑：`4 shards × 2 replicas`
- 写入迁移：`Vector` 双写
- 应用改造：自动建表能力开关化，目标集群必须关闭

我会把这三件事看成一体，而不是三个独立优化点。因为它们共同解决的是同一个问题：**如何让应用继续写熟悉的逻辑表名，同时把物理存储从单点平移到多分片多副本集群，而不要求业务代码理解 shard。**

## 为什么我会直接选 `4 × 2`

当前数据量是 `7118 GiB`。如果只做 `2 shards × 2 replicas`，每个 shard 的一份完整数据就已经接近 `3559 GiB`。在 ClickHouse 里，这个数字不能只拿来算“静态落盘”，还得把 merge、mutation、回灌放大、临时 part、失败重试和迁移期的新增写入一起算进去。

也就是说，`2 × 2` 不是不能用，而是太容易把后面的每一步都逼进很窄的资源走廊里。我会更愿意一开始就上 `4 × 2`，把单 shard 的静态数据量压到大约 `1779.5 GiB`，给迁移期间最容易出问题的几个环节留缓冲：

- 回灌时大批量 `INSERT SELECT` 形成的新 part；
- 后台 merge 对磁盘和 CPU 的持续消耗；
- `Vector` 双写期间目标集群同时承接新流量；
- 切换前后必须保留的新旧集群并行观察期。

如果让我用一句更直白的话概括，就是：**`4 × 2` 不是为了追求架构“好看”，而是为了把迁移过程中的资源压力变成可控问题。**

## 目标库应该长什么样

我不建议在目标侧保留“应用直写本地表”的模型，而是建议把目标集群一次建成最终形态：

- 数据库层使用 `Replicated` 数据库引擎；
- 本地表改成 `scalar_local`、`media_local`、`log_local`，引擎用 `ReplicatedMergeTree`；
- 对外仍然保留 `app.scalar`、`app.media`、`app.log` 这三个业务已经依赖的表名，但它们在目标侧应当变成 `Distributed` 表；
- 分片键优先使用 `cityHash64(projectId)`；
- 分区键建议补上，优先按 `createdAt` 做月分区。

我很看重这里的“逻辑表名保持不变”。因为当前应用查询和写入都直接绑定 `app.scalar`、`app.media`、`app.log`。如果迁移时顺手把表名和访问模型也改了，风险会从数据库层一路外溢到应用层、配置层和观测层。相反，如果让这些逻辑表名继续存在，只把它们背后的物理实现换成 `Distributed -> ReplicatedMergeTree`，业务侧的改动面会小很多。

## 为什么分片键应该先围绕 `projectId`

我更倾向把分片键定成 `cityHash64(projectId)`，而不是 `uid` 或更复杂的组合键。

原因不是抽象的“项目是业务主实体”，而是当前读路径已经把答案写得很清楚了：大多数查询都是先按 `projectId + experimentId` 过滤，再做聚合、取最新值、拉日志、拼媒体。也就是说，对这套业务来说，`projectId` 不是一个普通字段，而是**绝大多数查询能否局部命中的第一层边界**。

这会带来两个实际收益：

- 迁移后的项目级查询更容易在单 shard 内完成；
- 历史回灌时也更适合按 `projectId` 批次推进，减少“扫描一大圈、只取一点点”的无效工作。

当然，这个选择也不是永远正确。如果以后确认存在少数超大项目把某一 shard 压成热点，再考虑升级成 `cityHash64(projectId, experimentId)`。但在迁移阶段，我更反对为了“分布更漂亮”提前牺牲查询局部性。

## 这次迁移里，最危险的其实是自动建表

我觉得这是这套系统里最容易被低估的风险。

当前应用连接 ClickHouse 时会自动执行建表逻辑，而且建出来的还是裸 `MergeTree`。在旧单机时代，这个设计更像是一种开发便利；但一旦迁移到新集群，它就会从“方便”瞬间变成“污染目标 schema 的开关”。

我会把这块改成显式配置：

- 新增 `database.auto_create_tables`；
- 默认值可以保留 `true`，兼容本地开发和旧单机部署；
- 但新集群环境必须显式设成 `false`；
- 目标集群中的数据库和表结构必须由迁移 DDL 预建，而不是等应用连上去自己生成。

这是一个我会坚持先做掉的小改动。因为如果不先关掉这个隐患，后面所有关于“预建 `Distributed` 表”“统一逻辑表名”“避免错误本地表落库”的设计，都会变得很脆弱。

## 为什么我会用 `Vector` 双写，而不是赌一次停写回灌

当历史数据量已经是 `7118 GiB` 时，我对“短暂停写 + 一次补齐”这种说法会天然更警惕。不是因为理论上做不到，而是因为它把太多不确定性都压进了同一个窗口：

- 历史数据回灌到底要跑多久；
- 最后一轮增量补齐要扫多少数据；
- 源端扫描会不会反过来拖慢线上；
- 目标侧 merge 和复制会不会在切换前堆积。

既然当前写入主路径已经明确以 `Vector` 为准，那我更愿意把迁移设计成一场**先兜住新增写入，再慢慢搬历史**的工程。

具体来说，我会这样做：

1. 保留旧集群 sink 不动；
2. 在 `Vector` 里新增一套写往新集群的 sink；
3. 新旧两边都继续写相同的逻辑表名；
4. 双写开启的那个时刻记成迁移水位 `T0`；
5. 后续历史回灌只负责 `T0` 之前的数据；
6. 新写入从 `T0` 开始天然同时存在于新旧两边。

这件事真正重要的地方，不是“双写”这两个字本身，而是它把迁移问题拆成了两半：

- 新增写入的一致性，交给双写；
- 历史数据的完整性，交给分批回灌和校验。

我更信任这种分层，而不太信任“最后停机时一起解决”。

## 历史数据该怎么回灌

这次回灌我不建议按“整表一次扫完”或“纯按月份切批”来设计主路径。我会优先按 `projectId` 批次迁：

- 先拿到全量 `projectId` 清单；
- 按项目批次分组回灌；
- 对超大项目，再拆到 `experimentId` 粒度；
- 三张表分开迁，顺序优先 `scalar -> media -> log`。

我偏爱这个方案，是因为它和现有业务访问模式、主键顺序、未来分片键三者是一致的。也就是说，我不是单纯在找一个“可以分批”的维度，而是在找一个**读写路径和物理布局都共同认可的边界**。

这类回灌的典型形式应该是：

```sql
INSERT INTO app.scalar
SELECT *
FROM remote('old-clickhouse:9000', 'app', 'scalar', 'user', 'pass')
WHERE projectId IN ('p1', 'p2', 'p3')
  AND createdAt < toDateTime('2026-04-20 12:00:00');
```

其中 `createdAt < T0` 只是为了把历史和双写后的新增数据切开，真正的主批次边界还是 `projectId`。

## 迁移步骤我会怎么排

### 第一阶段：准备目标集群

- 搭建 `4 shards × 2 replicas + 3 Keeper`；
- ClickHouse 版本先和旧集群保持一致，不把“迁移拓扑”和“升级版本”绑在一起；
- 预建 `Replicated` 数据库、`*_local` 本地表和 `Distributed` 逻辑表。

### 第二阶段：做应用和写入管道改造

- 发布 `database.auto_create_tables` 开关；
- 在新集群环境中强制关闭自动建表；
- 为 `Vector` 增加一套写往新集群的 sink；
- 先做 shadow write，再切到正式双写。

### 第三阶段：冻结迁移水位 `T0`

- 正式开启双写；
- 记录 `T0`；
- 确保 `T0` 之后的新写入都同时进新旧集群。

### 第四阶段：长时间历史回灌

- 以 `projectId` 为主批次维度回灌 `T0` 之前的数据；
- 先迁 `scalar`，再迁 `media`，最后迁 `log`；
- 每一批都做计数和抽样校验；
- 对热点项目做更严格的对账。

### 第五阶段：切读不切写

- 先把读请求切到新集群；
- 继续保留双写；
- 观察一段时间，确认新集群的查询结果、复制状态和后台负载稳定。

### 第六阶段：下线旧写入

- 停掉旧集群 sink；
- 保留旧集群只读观察期；
- 观察通过后，再安排最终退役。

## 这套方案已经被本地跑通过什么

到这里，如果我还只是停留在“方案看起来合理”，我其实不会放心。对 `7118 GiB` 级别的数据迁移来说，最需要的不是再多一层抽象论证，而是先把最关键的工程链路在本地缩小规模后真正跑通。

这次我在本地做的验证，不是复刻云上旧集群，而是有意识地把验证范围收敛到迁移里最关键的闭环：

- 用一个本地 mock 单节点 ClickHouse 充当迁移源；
- 用 ClickHouse operator 起一套 `4 shards × 2 replicas + 3 Keeper` 的目标集群；
- 目标侧直接预建最终 schema：`app` 使用 `Replicated` 数据库，`scalar_local`、`media_local`、`log_local` 使用 `ReplicatedMergeTree`，`app.scalar`、`app.media`、`app.log` 对外保留为 `Distributed`；
- 在 `Vector` 中同时写旧源和新集群，拿它来模拟正式双写；
- 记录双写开启时刻为 `T0`，然后只回灌 `createdAt < T0` 的历史数据。

我很看重这个验证方式，因为它测试的不是某一条 SQL 能不能执行，而是这套生产方案最核心的四个承诺能不能同时成立：

- 目标集群能否按最终物理形态落库，而不是退回单机思维；
- 逻辑表名能否保持不变，让应用和写入链路不需要理解 shard；
- 新增写入能否通过 `Vector` 双写自然进入两边；
- 历史数据能否在双写持续开启的情况下按水位与批次安全追平。

## 本地验证得到的结论

这次验证最后跑出来的结果，已经足够让我把这条路从“建议方案”升级为“已验证路径”：

- `Vector` 双写成功把三张表的实时写入同时送到源端和目标端；
- 历史数据可以通过目标侧回拉源端的方式补齐到 `T0` 之前；
- 目标侧 `Distributed -> ReplicatedMergeTree` 的最终表形态可以稳定承接这套数据；
- 关闭应用自动建表后，应用仍然可以正常连接并使用预建好的目标 schema。

本地回灌完成后的核心对账结果是：

- `app.scalar`：`count() = 401`，`sum(uid) = 488801`
- `app.media`：`count() = 121`，`sum(uid) = 256142`
- `app.log`：`count() = 161`，`sum(uid) = 501723`

这些数字在源端和目标端保持一致。双写阶段我还额外插入了一组 `proj-live / exp-live` 的实时数据，三张表在源端和目标端都能查到 `1` 条，对我来说，这个信号很关键，因为它证明了这次迁移不只是“历史导入成功”，而是**历史回灌和实时新增写入可以同时成立**。

副本健康检查也通过了。目标集群 8 个 ClickHouse 节点上的三张本地表都满足：

- `total_replicas = 2`
- `active_replicas = 2`
- `queue_size = 0`
- `absolute_delay = 0`

这说明在这次验证规模下，复制并没有因为回灌和双写叠加而积压。换句话说，`4 × 2` 这个目标拓扑至少已经证明自己不是纸上设计，而是能把这三类核心表真实承接起来的结构。

## 本地验证暴露出的两个真实细节

真正有价值的本地验证，往往不是告诉我“一切正常”，而是尽早暴露那些只会在真实环境里出现的细小但致命的坑。这次最值得记下来的有两个。

第一，ClickHouse `24.3` 下的 `Replicated` 数据库引擎需要实验开关，但这个开关在我这次 `operator + 24.3` 的本地组合里，不能直接按顶层 server config 去塞。那样做会导致副本启动失败，因为它实际是 user-level setting。这个问题如果不先踩到，到了正式建库时会很容易把故障误判成 operator 配置或 Keeper 异常。

第二，在我这次本地 `24.3 + operator` 组合里，目标 schema 最后是通过逐节点执行 DDL 稳定建成的，而不是完全依赖 `ON CLUSTER`。我不会把这件事过度上升成“生产一定不能用 `ON CLUSTER`”，但我会把它当成一个很现实的提醒：**正式迁移前，必须先在与你的版本和 operator 配置一致的环境里验证建库建表方式，不要把 DDL 分发语义当成理所当然。**

后来我又把这条路径，在目标端升级到 ClickHouse `26.3` 后重新跑了一遍。这个重跑结果把我之前的判断又往前推了一步：**问题的关键已经不再是“Replicated 数据库是不是实验特性”，而是“在新版里，到底该把 `ON CLUSTER` 放在哪一层”。**

这次 `26.3` 的本地结果很明确：

- `Replicated` 数据库本身已经不再依赖实验开关；
- `CREATE DATABASE ... ENGINE = Replicated` 这一步仍然可以用 `ON CLUSTER`；
- 但数据库一旦建好，在这个 `Replicated` 数据库里继续对表执行 `ON CLUSTER`，会直接报错：`ON CLUSTER is not allowed for Replicated database`；
- 因此新版主路径应该改成“建库时 `ON CLUSTER`，建表时单点初始执行，再由 `Replicated` 数据库自动传播元数据”。

我很喜欢这个结果，因为它其实把职责边界讲得更清楚了。`ON CLUSTER` 负责的是“把数据库本身布出去”，而 `Replicated` 数据库负责的是“把数据库内部的表级元数据同步出去”。这两层如果混在一起，就容易把 DDL 策略搞成一种模糊的“哪里都试试看”；但一旦分清楚，执行路径反而比 `24.3` 时代更干净。

这次 `26.3` 重跑后的对账结果也依然成立：

- `app.scalar`：`count() = 401`，`sum(uid) = 568601`
- `app.media`：`count() = 121`，`sum(uid) = 335942`
- `app.log`：`count() = 161`，`sum(uid) = 581523`

这些数字和源端保持一致。目标集群 8 个 ClickHouse 实例全部运行在 `26.3.9.8`，而且 8 个实例都具备 `scalar/media/log` 与对应的 `*_local` 表，所有副本继续保持 `total_replicas=2`、`active_replicas=2`、`queue_size=0`、`absolute_delay=0`。对我来说，这比单纯说“升级到新版也能跑”更重要，因为它证明了**新版并不是让迁移路径失效，而是让 DDL 纪律变得更明确。**

## 为什么这次本地验证足以支撑生产方案

我不会因为本地数据量远小于生产就否认这次验证的价值。它当然不能替代生产容量压测，也不能替代云上网络、磁盘吞吐和长时间 merge 观察，但它已经验证了这次迁移最重要的结构正确性：

- 目标拓扑 `4 × 2` 是可建、可写、可复制、可查询的；
- `Vector` 双写这条实时迁移路径是可工作的；
- `T0 + 历史回灌` 这套问题拆分方式是成立的；
- 自动建表开关化后，应用能够服从“目标 schema 必须预建”的纪律。

对我来说，这意味着后续真正留给生产环境去解决的，已经主要是规模问题和节奏问题，而不再是“这条路本身是否成立”。这两类问题依然难，但它们至少是我愿意在生产里面对的难，而不是那种上线当天才发现迁移模型站不住的难。

## 这套方案里我最在意的几个前置条件

- 必须先拿到全量 `projectId` 清单，否则按项目批次迁移会很别扭；
- 必须先能关闭自动建表，否则目标 schema 会有被旧逻辑覆盖的风险；
- 必须让 `Vector` 双写具备足够的磁盘 buffer 和失败重试能力，否则双写会把新集群抖动放大成写入事故；
- 必须接受“迁移是一个持续过程”，而不是期待一次深夜窗口把所有事情做完。

## 我会怎么判断这次迁移已经足够安全

对我来说，真正的切换信号不是“回灌脚本都跑完了”，而是下面这些条件同时成立：

- 三张表的新旧两边总行数和关键聚合结果一致；
- 热门项目的业务查询结果一致；
- 新集群 `system.replicas` 没有复制堆积；
- 新集群后台 merge 没有把资源顶满；
- `Vector` 新 sink 的失败率、延迟和 buffer 占用已经稳定；
- 读流量切到新集群后，没有出现明显的长尾延迟和查询异常。

我更愿意把这些条件看成“能不能切”的真正标准，而不是把迁移是否完成定义成某条脚本退出码是否为零。

## 对我的提醒

这次方案最值得我记住的一点，是规模一旦到 7 TiB 左右，ClickHouse 迁移就不再是“数据库 DDL 设计题”，而是一个跨越**表结构、写入入口、回灌节奏、资源余量和切换纪律**的联合工程。也正因为如此，我更反对把版本升级、分片设计重构、去重语义调整和迁集群放在同一轮里一起做。迁移本身已经够难了，最好别再顺手往里塞别的愿望。
